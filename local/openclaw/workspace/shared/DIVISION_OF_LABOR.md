# 互补分工协议 — 本地 vs VPS

> 版本: v1.0 (2026-03-11)
> 制定依据: 两套龙虾帝国打通后的职责边界设计

---

## 核心原则

| 维度 | 本地 Mac（本部） | VPS 云端（分部） |
|------|----------------|----------------|
| 前提假设 | Leon **在场**，电脑开着 | Leon **不在场**，全天候自主 |
| 信任级别 | 高（可访问 Leon 私密数据） | 中（公开信息 + 指定数据源） |
| 响应模式 | 交互式、实时 | 异步、定时、批量 |
| 网络依赖 | 依赖 Leon 的网络 | 独立稳定的 VPS 网络 |

---

## 什么任务走本地

**条件满足其一即走本地：**

- 需要访问 **本地文件**（代码库、Obsidian、Desktop 文件）
- 需要 **Leon 实时确认**（工具权限审批、敏感操作）
- 需要操控 **本地工具**（浏览器、桌面 App、Claude Code CLI）
- 涉及 **私密/敏感信息**（健康数据、财务、个人笔记）
- 任务**短平快**，不超过 Leon 的单次会话

**负责角色（本地）：**
- 效率虾🦞：Leon 实时交互、文件操作、日常执行
- 架构虾🧬：系统设计、ADR、本地巡检
- 外包虾⚙️：需要 Lobster 工作流的复杂本地任务
- 管理虾🧠：项目/团队信息管理（需要访问内部数据源时）

---

## 什么任务走 VPS

**条件满足其一即走 VPS：**

- 需要在 **Leon 离线时**（睡觉、出门、电脑关着）执行
- 是**定时重复任务**（每天/每周，不依赖 Leon 在场）
- 是**耗时批量任务**（不值得占用本地资源等待）
- 依赖**稳定的公网出口**（爬取、API 调用、webhook 接收）
- 任务结果**下一天才需要**（夜间批处理）

**负责角色（VPS）：**
- 自动虾🤖：定时任务、批量执行、跨节点 inbox 接收
- 养生虾🌿：健康监测、提醒（个人健康数据在此隔离）
- 情报虾🔍：持续信息收集、行业监控（Kimi K2.5）
- 编程虾💻：云端代码运行、构建、部署相关任务

## 任务路由决策树

```
收到一个任务
    │
    ├─ 需要本地文件/工具？ → 本地
    ├─ 需要 Leon 实时确认？ → 本地
    ├─ Leon 马上要看结果？ → 本地
    │
    ├─ 定时重复？ → VPS
    ├─ Leon 不在时执行？ → VPS
    ├─ 耗时 > 30分钟？ → VPS（投递到自动虾 inbox）
    └─ 纯公网信息获取？ → VPS（情报虾）
```

---

## 跨节点投递方式（kuro → VPS 自动虾）

当本地 kuro 判断某任务应走 VPS，通过以下方式投递：

```bash
# 方式一：创建 VPS 一次性 cron（立即执行）
vps-claw cron create --agent auto --message "任务内容" --at "now"

# 方式二：写入文件投递（自动虾定期轮询）
# 写入 /root/.openclaw/workspace/auto/inbox/task-YYYYMMDD-XXX.yaml
```

---

## 不适合跨节点的任务

- 需要 **双向实时交互** 的任务（不要跨节点，走本地）
- 涉及 **健康/隐私数据** 的任务（只走养生虾，不路由到其他 VPS agent）
- **紧急任务**（用 Slack 直接 @ 对应 agent，不用 cron/inbox）
